**File 1: scripts/parse_saveastext.py -- SaveAsText Parser**

Build a structured parser for SaveAsText output files. The format is hierarchical BEGIN/END blocks with property=value pairs.

The parser must extract:

1. **Top-level properties:** RecordSource, Caption, DefaultView, Filter, OrderBy, OnOpen, OnClose, OnCurrent, AllowAdditions, AllowDeletions, AllowEdits
2. **Controls:** For each control block (Begin TextBox, Begin ComboBox, Begin CommandButton, Begin Subform, Begin ListBox, Begin Label, Begin CheckBox, Begin OptionButton, Begin Image, Begin TabControl, Begin OptionGroup):
   - Name, ControlSource, RowSource, RowSourceType, Caption, DefaultValue, ValidationRule, InputMask, Visible, Enabled, Locked
   - ALL event properties: OnClick, OnDblClick, OnOpen, OnClose, BeforeUpdate, AfterUpdate, OnChange, OnGotFocus, OnLostFocus, OnEnter, OnExit (value will be `[Event Procedure]` if VBA handler exists)
3. **Sections:** Detail, FormHeader, FormFooter, PageHeader, PageFooter -- with their controls
4. **Code-behind VBA:** Everything after the `CodeBehindForm` marker. Extract the full VBA code text. Identify individual Sub/Function names and their event associations (e.g., `Private Sub cmdSave_Click()` -> control "cmdSave", event "Click").
5. **Subform references:** For Subform controls, extract SourceObject property (the linked form/report name)

**Parser approach:** Use recursive descent for BEGIN/END blocks. Do NOT use ad-hoc regex for the block structure (per research anti-patterns). Simple regex is fine for property extraction within a block.

**Encoding handling:** Try reading files in order: utf-16-le (SaveAsText default on many systems), utf-8, cp874, cp1252. Use `errors="replace"` as final fallback.

Expose two functions:
- `parse_form(filepath: Path) -> dict` -- returns structured form data
- `parse_report(filepath: Path) -> dict` -- returns structured report data (same format, different top-level properties)

**File 2: scripts/extract_forms_reports.py -- Assessment Document Generator**

This script reads all parsed form/report data plus the query JSON from Plan 03-01 and generates the assessment documents.

1. **Load inputs:**
   - Parse all .txt files in windows/export/forms/ using parse_saveastext.py
   - Parse all .txt files in windows/export/reports/ using parse_saveastext.py
   - Load assessment/queries/_raw_queries.json for query SQL cross-reference
   - Load assessment/relationships.md for table relationship context

2. **Generate assessment/forms/_overview.md:**
   - **Summary section:** Total forms, forms with code-behind, forms with subforms
   - **Form Catalogue table** (tabulate pipe-format): Name, Caption (Thai), Record Source, Control Count, Has Code-Behind?, Has Subforms?, Key Events
   - **Per-form detail sections** (within the same file, as ### headers): For each form, list:
     - Purpose (inferred from caption + record source + controls)
     - Record Source (table or query name)
     - Key controls with data bindings (name, type, control source)
     - Event handlers (control name, event, handler type)
     - VBA code-behind (if any) -- include the full code text in a ```vba code block
     - Subforms referenced

3. **Generate assessment/forms/navigation.md:**
   - Analyze all forms for navigation patterns:
     - CommandButton controls with OnClick events that reference `DoCmd.OpenForm` or `DoCmd.OpenReport` in VBA code-behind
     - The main form (หน้าหลัก / "Main Page") likely has navigation buttons
     - Subform references (SourceObject properties)
   - Produce a text-based navigation tree showing: Main Menu -> Form A -> Subform B, Main Menu -> Report C
   - Include a Mermaid diagram: `graph TD` with form nodes and navigation edges
   - If no VBA navigation is found (possible in a simple system), document the navigation based on subform nesting

4. **Generate assessment/reports/_overview.md:**
   - **Summary section:** Total reports, reports with subreports, reports with code-behind
   - **Report Catalogue table** (tabulate pipe-format): Name, Caption (Thai), Record Source, Control Count, Has Code-Behind?, Output Type (invoice, list, label)
   - **Per-report detail sections:** For each report:
     - Purpose (inferred from name + record source)
     - Record Source (table or query)
     - Key fields displayed (control sources from Detail section)
     - Grouping/sorting (from GroupHeader/GroupFooter sections if present)
     - Calculated fields (controls with expressions in ControlSource starting with "=")

Use tabulate pipe-format for ALL tables (per project convention).

**File 1: assessment/business_logic/process_flows.md**

Document the complete business processes by tracing data flow across components:

1. **Order Management Flow:**
   - Trace: Order entry forms (frm_salesorder_fishingshop, frm_salesorder_retail) -> order tables (รายละเอียดออเดอร์, สินค้าในแต่ละออเดอร์) -> invoice reports (ใบกำกับภาษี*)
   - Document: What happens when a user creates an order? Which form fields map to which table columns? Which queries calculate order totals? Which reports print the invoice?

2. **Inventory Flow:**
   - Trace: Goods receipt form (frm รับเข้าสินค้า) -> receipt tables (หัวใบรับเข้า, สินค้าในแต่ละใบรับเข้า) -> stock queries -> stock form (frm_สต็อคสินค้า)
   - Trace: Goods issue form (frm เบิกสินค้า) -> issue tables (หัวใบเบิก, สินค้าในแต่ละใบเบิก) -> stock queries
   - Document: How is stock tracked? What queries calculate current stock levels?

3. **Customer Management Flow:**
   - Trace: Member form (frmข้อมูลสมาชิก) -> member table (ข้อมูลสมาชิก) -> points queries/tables
   - Document: How are customers registered? How does the points system work?

4. **Reporting Flow:**
   - Map each report to its data source (query or table) and the form that likely opens it
   - Document: Sales tax reports, shipping documents, address labels -- what data feeds each?

For each flow, produce:
- **Text description** in English explaining the business process step by step
- **Component map** showing: Form -> Table -> Query -> Report connections
- **Cross-references** to specific assessment documents (link to query overview, form overview, table docs)

**File 2: assessment/business_logic/pricing_discounts.md**

Search for pricing and discount logic across all extracted components:

1. **Query SQL analysis:** Look for IIF(), Switch(), calculated columns involving price * quantity, discount percentages, tax calculations (VAT is 7% in Thailand)
2. **Form code-behind analysis:** Look for VBA code that calculates totals, applies discounts, or validates pricing
3. **Table constraints:** Check validation rules on price/amount columns from the schema docs
4. **Points system:** Document the customer points calculation and redemption logic from the คะแนน-related queries and tables

If specific pricing/discount logic is NOT found (possible in a simple system where pricing is just stored values with no calculation logic), document that clearly: "No programmatic pricing/discount logic found. Prices appear to be stored as fixed values in [table]. Discounts, if any, are applied manually."

For inventory/formula logic (VBA-03 requirement): Check if any queries or VBA code map raw materials to finished products. If not found, document the absence.

Use tabulate pipe-format for any tables. Cross-reference all claims to specific queries/forms/tables by name.
