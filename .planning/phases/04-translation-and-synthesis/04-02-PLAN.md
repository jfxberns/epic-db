## Risks, Anti-Patterns, and Improvement Opportunities

This is a DEDICATED section (not per-component) per locked decision. Consolidate all risks, anti-patterns, and improvement suggestions found throughout the assessment. Organize as:

### Data Integrity Risks
- ALL foreign keys use NO_INTEGRITY (grbit=0x00000002) -- no cascade, no restrict
- Stock is calculated, never stored -- calculation depends on 3 sub-queries being correct
- Payment status tracked via free-text field (ธนาคาร) -- no enum constraint
- Points system uses 3 fixed columns instead of a transaction log

### Structural Anti-Patterns
- คะแนนที่ลูกค้าใช้ไป (customer_points_used) table is abandoned (0 rows, no relationships)
- Dual customer tables (shop vs retail) with no shared base -- creates duplication
- Order details table stores both shop and retail orders in one table, distinguished by channel field
- Invoice numbers managed via query-based counter (qryกำหนดเลขที่inv) -- race condition risk
- Some queries use [Forms]! parameter references making them unusable outside their form context

### Missing Features / Gaps
- 4 corrupt forms (sales order forms for both channels, 2 stock forms) -- core data entry path is damaged
- No audit trail -- no created_at/updated_at timestamps on most tables
- No user authentication or access control
- No backup/export mechanism

### Improvement Opportunities for Rebuild
- Replace calculated stock with stored balance + transaction log
- Replace 3-column points system with points_transactions table
- Add proper enum/lookup tables for payment status, order channel, unit types
- Add referential integrity constraints
- Add audit timestamps (created_at, updated_at, created_by)
- Normalize dual customer types into single customers table with type flag
- Separate shop/retail order workflows into distinct but linked tables if business rules differ significantly

For each item: describe what it is, why it matters, and what the rebuild should do differently.
