**Step 4: Create scripts/extract_query_sql.py.**

The script must:
1. Start JVM with Jackcess JARs on classpath (using `jpype.startJVM(classpath=[...])`)
2. Open epic_db.accdb via `DatabaseBuilder.open(File(str(DB_PATH)))` in READ-ONLY mode
3. Iterate `db.getQueries()` and call `query.toSQLString()` for each
4. Extract: name, type (from `query.getType()` -- the Java enum, not MSysObjects flags), SQL text, hidden flag
5. Include the 8 hidden `~sq_c*` subqueries in extraction (do NOT filter them out -- they are subform data sources per Pitfall 7 in research)
6. Handle errors gracefully: if `toSQLString()` throws, capture the error message but continue
7. Follow the established pattern: DB_PATH = Path(__file__).resolve().parent.parent / "data" / "epic_db.accdb", LIB_DIR = Path(__file__).resolve().parent.parent / "lib"
8. Output JSON to stdout AND save to assessment/queries/_raw_queries.json for downstream use
9. Print a summary table (using tabulate pipe-format) showing: query name, type, SQL length, error status

**IMPORTANT fallback:** If JPype/Jackcess fails at runtime (JVM startup fails, database cannot be opened, or toSQLString crashes), do NOT spend more than 10 minutes debugging. Instead:
- Document the failure in a comment in the script
- Write a stub that outputs an error message explaining queries will be extracted on Windows in Plan 03-02
- The Windows export_all.vbs (Plan 03-02) already extracts QueryDef.SQL as a backup path

**Bracket-quoted Thai identifiers:** The SQL will contain `[สินค้า]`, `[ข้อมูลร้านค้า]` etc. These are normal Access SQL bracket notation. The extraction script must preserve them exactly as-is (no encoding conversion needed -- Jackcess reads UTF-16 from the binary format).

**Prerequisite:** assessment/queries/_raw_queries.json must exist from Task 1. If Task 1 failed completely (Jackcess fallback triggered), skip this task -- the query documents will be produced after Windows extraction in Plan 03-03.

**Step 1: Build reference sets for dependency analysis.**

Load the list of known user tables from db_reader.py's `get_user_tables()` (10 tables). Load the list of known query names from the raw JSON. These two sets are needed to classify references in SQL as "table reference" vs "query reference" vs "unknown."

**Step 2: For each query, parse the SQL to extract:**
- **Type classification** from actual SQL (not MSysObjects flags):
  - Starts with SELECT -> SELECT
  - Starts with UPDATE -> UPDATE
  - Starts with INSERT -> INSERT
  - Starts with DELETE -> DELETE
  - Contains TRANSFORM -> CROSSTAB
  - Contains UNION -> UNION
  - Contains PARAMETERS -> mark as PARAMETERIZED (in addition to base type)
  - Detect `[prompt text]` patterns -> mark parameter prompts
- **Table references:** Extract identifiers after FROM, JOIN, INTO, UPDATE keywords. Match against known tables.
- **Query references:** Same extraction, match against known query names (including `~sq_c*` subqueries).
- **Purpose annotation:** Generate a 1-line English purpose description from: (a) the query name (translate Thai to English), (b) the tables referenced, (c) the type of operation. Example: "SELECT from สินค้า joined with สินค้าในแต่ละออเดอร์ -- retrieves product details per order."

**Step 3: Generate assessment/queries/_overview.md.**

Format: Markdown document with tabulate pipe-format tables.

Sections:
1. **Summary** -- total queries by type, count of parameterized queries, count of hidden subqueries
2. **Query Catalogue** -- table with columns: Name, Type, Tables Referenced, Queries Referenced, Parameterized?, Hidden?, Purpose
3. **Hidden Subqueries** -- separate table for the `~sq_c*` queries, noting which forms likely use them
4. **Query Type Distribution** -- counts by type

**Step 4: Generate assessment/queries/dependency_graph.md.**

Format: Markdown document with:
1. **Mermaid graph** -- `graph LR` showing table nodes (rectangles) -> query nodes (rounded). Use `safe_id()` function to convert Thai names to hash-based IDs. Include legend.
2. **Dependency table** -- tabulate pipe-format table: Query Name, Depends On (tables), Depends On (queries), Depended On By (queries)
3. **Orphan queries** -- queries that reference no known table or query (possible errors or parameter-only queries)
4. **Hub tables** -- tables referenced by the most queries (these are the core data tables)

Use tabulate pipe-format for all tables (per project convention).
