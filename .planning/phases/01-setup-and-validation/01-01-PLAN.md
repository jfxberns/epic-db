<task type="auto">
  <name>Task 1: Install access-parser and create db_reader module</name>
  <files>scripts/db_reader.py</files>
  <action>
1. Install access-parser into the project venv:
   ```
   uv pip install access-parser tabulate
   ```
   If access-parser fails to install or fails at runtime on the .accdb file (e.g., version incompatibility per Research pitfall #5), try the McSash fork as fallback:
   ```
   uv pip install git+https://github.com/McSash/access_parser_c
   ```

2. Create `scripts/db_reader.py` with these functions:
   - `DB_PATH` constant pointing to `data/epic_db.accdb` (use `Path(__file__).parent.parent / "data" / "epic_db.accdb"`)
   - `open_db(db_path=DB_PATH)` — returns an AccessParser instance. Wrap in try/except to catch and report version errors clearly.
   - `get_catalog(db)` — returns `db.catalog` (the list of user-visible table names)
   - `parse_table(db, table_name)` — wraps `db.parse_table(table_name)`, returns the dict of column_name -> values
   - `get_table_names_from_msys(db)` — reads MSysObjects, filters for Type==1 (local tables), excludes MSys* prefixes, returns list of user table names
   - `get_all_objects_from_msys(db)` — reads MSysObjects, returns dict grouped by type using the MSYS_TYPES mapping from research:
     ```python
     MSYS_TYPES = {
         1: "Table (Local)",
         4: "Table (Linked ODBC)",
         5: "Query",
         6: "Table (Linked)",
         -32768: "Form",
         -32764: "Report",
         -32766: "Macro",
         -32761: "Module",
     }
     ```
     For each object, store name and type. Return structure: `{"Table (Local)": ["name1", ...], "Query": [...], ...}`

3. Add a `if __name__ == "__main__"` block that:
   - Opens the database
   - Prints the catalog (user table names)
   - Prints one line per MSysObjects type with count
   - Reads the first table from the catalog and prints first 3 rows as a formatted table (using tabulate)
   This serves as a quick smoke test.

4. Run `python scripts/db_reader.py` to verify it works. If it fails:
   - On "database version not supported": try McSash fork
   - On import errors: check that pip install succeeded in the right venv
   - On file not found: verify DB_PATH resolution
   Capture and report the output.

<task type="auto">
  <name>Task 2: Validate Thai text encoding in table data and field names</name>
  <files>scripts/validate_encoding.py</files>
  <action>
1. Create `scripts/validate_encoding.py` that imports from db_reader and performs comprehensive Thai encoding validation:

   a. **Field/column name check**: For every table in the catalog, get column names and check each for Thai characters (Unicode range 0x0E00-0x0E7F). Print all column names that contain Thai, grouped by table.

   b. **Row data check**: For every table in the catalog, read all rows and scan every string value for:
      - Thai characters present (good)
      - Replacement character U+FFFD (mojibake indicator)
      - Unexpected high codepoints or control characters (corruption indicator)
      Print a summary per table: total string values scanned, count containing Thai, count with suspected mojibake.

   c. **Visual output for spot-checking**: For each table that contains Thai data, print 3-5 sample values that contain Thai text, formatted clearly with the table name, column name, and the actual Thai string. This output is what the user will visually verify (or get help verifying).

   d. **Summary verdict**: At the end, print a clear PASS/FAIL verdict:
      - PASS: Thai characters found in data, zero mojibake indicators detected
      - FAIL: Mojibake detected — print which tables/columns are affected and suggest trying McSash fork
      - WARNING: No Thai characters found at all — something unexpected, flag for user review

2. Run the script. Capture full output.

3. **If encoding is BROKEN** (FAIL verdict):
   - Install the McSash fork: `uv pip install git+https://github.com/McSash/access_parser_c`
   - Update db_reader.py import to use access_parser_c instead of access_parser
   - Re-run validate_encoding.py
   - If still broken, document the failure clearly — this becomes a blocker per user decision

4. **If encoding PASSES**: No further action needed. The output will be included in the plan summary.
  </action>
  <verify>
Run `python scripts/validate_encoding.py` from project root. Must see:
1. Column name analysis (which names contain Thai)
2. Per-table encoding summary (string values scanned, Thai count, mojibake count)
3. Sample Thai text values printed for visual verification
4. Clear PASS/FAIL/WARNING verdict
No Python exceptions.
  </verify>
  <done>
Thai encoding validation script exists and has been run against the full database. Verdict is PASS (Thai renders correctly in both column names and row data) OR the issue has been resolved by switching to McSash fork OR the failure is documented as a blocker.   </done>
